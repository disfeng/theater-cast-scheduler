# 剧场卡司排班

> 面向剧场运营的开源卡司排班系统：管理场次、角色、演员能力与请假，导入指定和许愿，并生成可解释的周排班预览。

这个项目把剧场基础资料、月度场次、演员请假、玩家指定/许愿与排班规则放到同一条工作流中。它不只输出“谁演哪个角色”，也会保留空槽和未满足指定的失败原因，方便排班员复核与人工决策。

## 功能总览

- **基础资料**：多剧场、每日场次、默认周模板、剧场角色、演员能力与个人排班限制。
- **月度计划**：在可视化日历中按周模板生成和调整场次，支持闭店日、月份切换与状态区分。
- **指定与许愿**：群文本解析、人工校正、规则校验、幂等确认和周批次锁定。
- **权益管理**：按剧场配置通用或指定类道具，支持批量发放、独立有效期、玩家背包、手工核销与全量流水。
- **排班引擎**：硬规则过滤、三级指定优先级、许愿偏好与失败原因输出。
- **双角色入口**：超级管理员和剧场管理员负责配置、审核与日志追溯；演员通过移动端查看已披露排期、指定和对位玩家，并提交请假申请。

## 项目解决什么问题

剧场排班往往同时受到角色能力、请假、连场上限、玩家付费指定、许愿和演员工作量的影响。当这些信息散落在群消息、表格和个人记忆中时，手工排班容易出现：

- 请假、跨卡能力或连场限制被遗漏。
- 不同类型指定的优先级难以统一执行。
- 许愿与工作量均衡缺少一致的衡量方式。
- 导入数据一旦出错，很难追溯原文和修正过程。
- 排班失败时只看到结果，不知道具体违反了哪条规则。

本项目以“月度基础计划 + 周排班批次”为主线，将上述信息结构化，再用可测试、可解释的规则生成排班建议。

## 已实现功能

### 管理员端

- 管理多个剧场，为每个剧场独立配置每日演出场次和默认周模板。
- 管理剧场角色与角色分组，并从角色列表反向查看所有可出演演员。
- 管理演员评级、最大连场数、低评级月度上限和备注；支持按姓名、评级和剧场筛选。
- 按剧场分组配置演员的可出演角色，支持跨剧场角色能力。
- 在月度日历中生成并编辑演出场次，支持闭店日、按周模板恢复和已有计划状态展示。
- 查看和审核演员的整天请假申请。
- 在周排班矩阵中按“演出场次 × 角色”选择演员，推荐功能只补齐空位，不覆盖人工安排。
- 保存或发布前执行角色能力、请假、同场唯一、月度上限和跨周/月/年连场冲突检测；存在冲突时必须二次确认。
- 支持排班草稿、乐观版本校验与正式发布，未发布内容不会出现在演员端。
- 支持整周发布和单日发布；单日修改可独立重新发布，不影响同周其他日期的正式排班快照。
- 支持导出当前周内部排班 CSV；演员端可导出本人当月已披露班次 CSV。
- 支持超级管理员与剧场管理员的权限隔离，并通过审计日志记录关键操作。

### 指定与许愿导入

- 按月向玩家逐件发放万能、热力榜前三和对位指定权益；玩家背包展示独立来源、到期时间、状态与完整流水。
- 按具体演出场次保存信息板版本，支持确定性解析，以及可配置的 OpenAI 兼容 AI 解析服务和人工复核降级流程。
- 本人指定自动核对库存，代指定记录客服授权核对；确认后预占具体权益并锁定周排班单元格。
- 许愿绑定具体演出场次并作为软偏好参与推荐；发布时核销已满足指定，未满足或取消场次时释放权益。
- 提供按类型、来源月份、登记来源、玩家和状态分组的库存/流水核对，并可下钻到具体道具及流水。
- 为某个剧场的自然周创建唯一批次。
- 粘贴群统计文本，生成可追溯的导入草稿，不直接写入正式数据。
- 自动匹配演员与角色，无法确定的内容保留原文等待人工处理。
- 支持人工修正类型、玩家、演员、角色、目标场次和备注，修改后立即重新校验。
- 支持逐条确认和批量确认；重复确认不会重复生成正式记录。
- 批次进入 `ready` 后锁定导入内容，防止后续编辑与并发写入。

### 权益管理

- 权益数据按剧场隔离，进入页面后先选择剧场，道具配置、发放、背包与流水均只展示当前剧场数据。
- 支持自定义道具名称、类型、启用状态和默认有效期；每种道具可组合配置“限定持有人本人使用”和“发放时绑定演员”，覆盖自由使用、绑定演员、仅本人使用、本人使用且绑定演员四种模式。
- 权益发放分为“按玩家批量发放”和“按演员批量发放”：非演员绑定道具按玩家矩阵录入，演员绑定道具可同时选择多位演员并分别粘贴玩家名单；批次可保存草稿，确认后幂等发放。
- 玩家背包可按玩家、道具和状态查询，支持通用道具手工核销，并在提交前预览实际扣减的道具。
- 权益流水记录发放、预占、核销、退回和手工调整，可从背包直接跳转查看指定道具的完整记录。

### 排班引擎与 API

- 为场次与角色槽位筛选可用演员。
- 应用请假、角色能力、同场唯一和最大连场等硬规则。
- 按万能指定、榜单前三指定、对位指定的顺序尝试满足权益。
- 在剩余候选人中考虑许愿和月度工作量。
- 输出排班结果、未满足指定和空槽，供前端或其他系统继续处理。
- 提供管理员、演员、月度计划、导入和排班预览 API。

### 演员端

- 以手机号关联演员账号，首次登录强制修改初始密码，后续可在“我的”中自主修改密码。
- 工作台仅显示已发布且已到披露时间的排班，支持月历/列表视图，并展示角色、指定类型和对位玩家。
- 支持按剧场提交多日整天请假、查看逐日审批结果与撤回未审批申请。
- 所有演员端业务页都使用演员昵称、脱敏手机号和时间水印，降低排班截图外泄风险。
- 剧场可配置 N 天前的排班披露时间，并可选开启短信登录提醒。

## V1 工作流程

1. **基础配置**：建立剧场、自定义场次、默认周模板、固定角色和演员跨剧场角色能力。
2. **月度准备**：选择剧场和月份，在日历中生成或微调场次并标记闭店日，同时审核演员请假。
3. **周数据导入**：选择自然周，导入群统计文本或手工添加指定/许愿。
4. **复核确认**：修正解析失败的条目，逐条或批量确认有效数据，完成后将批次标记为 `ready`。
5. **周排班**：选择剧场与自然周，在“场次 × 角色”矩阵中人工安排或一键补齐空位，并复核冲突。
6. **保存与发布**：先保存草稿继续调整；发布前再次校验。管理员可发布整周，也可独立发布或重发某一天，正式快照到达剧场披露时间后才会出现在演员端。

## 排班规则

引擎按以下层级处理排班：

1. **硬规则**：已批准请假不可排班；演员必须会该角色；同一场不能出演两个角色；不得超过个人最大连场数；暂停演员不参与排班。
2. **指定优先级**：万能指定 > 榜单前三指定 > 对位指定。指定不能覆盖硬规则。
3. **补排偏好**：指定之后再考虑许愿、月度场次与排班密度。许愿只影响偏好，不会压过指定。
4. **可解释输出**：返回未满足指定的失败原因和尚未安排的空槽。

V1 使用可解释的分层规则与评分方式，不追求数学意义上的全局最优。

## 快速开始

### 环境要求

- Python 3.11+
- Node.js 18+ 与 npm
- MySQL 8.0

### 1. 创建数据库

```bash
mysql -u root -p -e "CREATE DATABASE theater_cast_scheduling CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"
```

### 日常一键启动

首次完成后端虚拟环境、前端依赖和根目录 `.env` 配置后，在仓库根目录执行：

```bash
./dev.sh
```

脚本会先运行数据库迁移，再同时启动后端与前端；按 `Ctrl+C` 会一并停止两个服务。如果依赖尚未安装，请先按下面的步骤分别初始化后端和前端。

### 2. 启动后端

在仓库根目录执行：

```bash
cd backend
python3 -m venv .venv
source .venv/bin/activate
pip install -e ".[dev]"

export DATABASE_URL="mysql+pymysql://root:password@localhost:3306/theater_cast_scheduling"
export JWT_SECRET="replace-with-a-random-secret"

alembic upgrade head
uvicorn app.main:app --port 7004 --reload
```

后端默认运行在 <http://localhost:7004>，可通过 <http://localhost:7004/health> 检查状态。

### 3. 启动前端

新建一个终端，回到仓库根目录后执行：

```bash
cd frontend
npm install
npm run dev
```

前端默认运行在 <http://localhost:7003>，API 客户端请求 <http://localhost:7004>。后端已允许 `localhost:7003` 和 `127.0.0.1:7003` 两个本地开发来源；部署到其他域名时，请改为实际前端来源。

### 生产部署基线

复制 `backend/.env.example` 和 `frontend/.env.example`，替换数据库账号、随机 JWT 密钥、前端域名、API 地址及演员端入口。生产环境必须设置：

```bash
APP_ENV=production
ALLOW_DEMO_ADMIN=false
CORS_ALLOWED_ORIGINS=https://scheduler.example.com
VITE_API_BASE_URL=https://api.scheduler.example.com
```

生产配置检查和启动：

```bash
./scripts/check-production-config.sh
./scripts/start-production.sh
```

`/health` 只检查进程存活；`/ready` 同时检查数据库连接和 Alembic 迁移是否处于当前 head。发布系统应在切换流量前确认 `/ready` 返回 200。

### 数据库备份、恢复与回滚

建议每日备份、保留至少 14 天，并在每次迁移前额外备份：

```bash
export MYSQL_HOST=127.0.0.1 MYSQL_PORT=3306
export MYSQL_DATABASE=theater_cast_scheduling MYSQL_USER=backup_user MYSQL_PASSWORD='replace-me'
./scripts/backup-mysql.sh
```

恢复必须明确指定备份文件和确认参数，建议先在独立测试库完成恢复演练：

```bash
MYSQL_DATABASE=theater_cast_restore ./scripts/restore-mysql.sh backups/theater_cast_scheduling-YYYYMMDD-HHMMSS.sql.gz --confirm
```

发布失败时先停止新流量并回滚应用版本；只有数据库变更无法向前兼容且已验证备份可恢复时，才执行数据库恢复。恢复后重新运行配置检查、`alembic current`、`/health` 和 `/ready`。

## API 文档

启动后端后可使用 FastAPI 自动文档：

- Swagger UI：<http://localhost:7004/docs>
- ReDoc：<http://localhost:7004/redoc>
- 健康检查：<http://localhost:7004/health>
- 就绪检查：<http://localhost:7004/ready>

主要路由前缀：

- `/auth`：登录与 Token。
- `/admin`：基础资料、权益、月度计划、请假审核、周排班、信息板、权限与审计日志。
- `/actor`：演员工作台、演出日历、个人排班、请假与账号安全。
- `/scheduling`：周排班预览。

```bash
backend/.venv/bin/ruff check backend
cd backend
pytest -q
```

后端测试默认使用 SQLite 内存数据库，因此不会修改本地 MySQL 数据。

### MySQL 查询健康检查

在已配置开发数据库的环境中，可执行只读的 `EXPLAIN` 巡检：

```bash
cd backend
.venv/bin/python -m app.cli.query_health
```

命令会检查月度场次、权益背包、权益流水、请假审批和演员排班等高频查询是否命中预期索引，并拒绝大表全表扫描或 filesort；它不会开启数据库全局慢查询日志，也不会修改数据库配置。应用会通过 `SLOW_QUERY_THRESHOLD_MS`（默认 500 毫秒，设为 0 可关闭）记录不含参数值的慢 SQL，生产环境仍建议由数据库运维侧同时启用实例级慢查询日志。
