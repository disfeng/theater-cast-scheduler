# 管理基础功能补全设计

## 背景与目标

现有实现已经提供剧场、角色、演员、请假审核和月度计划的后端接口，也提供了对应的前端入口，但管理页面仍以只读展示为主，月度计划页面无法发起生成。同时，整月重生成会无条件删除既有场次，周模板中的重复场次还可能触发数据库约束错误。

本次补全的目标是让 V1 管理基础真正可操作，并保证月度重生成不会破坏已排班或已发布数据。

## 范围

本次包含：

- 剧场与固定角色的新增和列表刷新。
- 演员新增，以及评级、最大连场、低评级月上限、备注和可出演角色的编辑。
- 月度计划的剧场、年月和闭店日期输入，以及生成后的场次刷新。
- 周模板的结构、场次值和重复值校验。
- 月度重生成的安全检查、事务回滚和明确错误响应。
- 对上述工作流的服务、API 和前端自动化测试。

本次不包含通用 CRUD 框架、多步骤向导、删除剧场/角色/演员、周排班编辑、发布和导出。

## 前端设计

### 基础配置页

页面采用内联表单和结果列表，不引入弹窗。剧场表单包含名称及周一至周日的早场、晚场选择，提交到现有剧场创建接口；角色表单包含名称和可选分组，提交到现有角色创建接口。保存成功后重新请求对应列表，失败时在页面显示错误信息。

### 演员管理页

页面顶部提供新增演员表单，字段为显示名、评级、最大连场、低评级月上限和备注。演员列表的每一行提供相同字段的编辑控件，并以角色复选框维护可出演角色。保存演员基础字段后调用演员更新接口，保存角色能力时调用能力替换接口；每次成功操作后刷新演员列表。

### 月度计划页

页面提供剧场选择、年份、月份和闭店日期输入。闭店日期使用逗号或换行分隔的 ISO 日期列表，前端在提交前去空白并过滤空值。点击生成后调用月度生成接口，并用响应替换当前场次列表。年月或剧场变化时读取对应月份场次；生成失败时显示后端错误。

## 后端设计

### 周模板校验

剧场创建请求只接受 `monday` 至 `sunday` 七个键，场次只接受 `early` 和 `late`。同一天的场次不可重复。非法输入由 Pydantic 返回 422，避免数据库唯一约束成为普通输入错误的第一道防线。

### 安全重生成

重生成前读取目标剧场和月份的现有场次：

- 不存在既有场次时，直接生成草稿。
- 全部为草稿且没有排班或指定引用时，删除旧草稿并重新生成。
- 存在非草稿场次、排班引用或指定引用时，拒绝重生成并返回 409，不修改任何数据。

服务层使用明确的领域异常表示冲突；路由层将其转换为 409。数据库提交失败时回滚事务，避免会话停留在失败状态。

### 演员能力更新原子性

替换能力前先验证演员及全部角色 ID。只有验证全部通过后才删除旧能力并写入新能力，从而避免调用方捕获异常后意外提交半完成变更。

## API 客户端补充

前端客户端增加演员更新和能力替换方法，并支持解析后端 `detail` 错误。现有创建、月度生成和请假审核方法继续复用统一的授权请求逻辑。

## 测试策略

后端测试覆盖：

- 非法星期、非法场次和重复场次返回 422。
- 纯草稿月份可以重生成。
- 非草稿或存在关联记录的月份返回 409，原数据保持不变。
- 无效角色 ID 不会删除演员原有能力。

前端测试覆盖：

- 剧场和角色表单提交正确载荷并刷新列表。
- 新增演员及编辑评级、限制和能力会调用正确接口。
- 月度计划提交选择的剧场、年月和闭店日期，并展示生成结果。
- API 失败时页面显示错误信息。

最终验证包括 Ruff、后端全量 pytest、前端全量 Vitest、TypeScript 和 Vite 生产构建，以及生成物未被 Git 跟踪检查。

## 验收标准

- 管理员无需直接调用 API 即可完成剧场、角色和演员基础维护。
- 管理员可以维护演员评级、连场限制、低评级月上限和可出演角色。
- 管理员可以选择剧场、年月和闭店日期并生成月度计划。
- 月度重生成不会删除或重置已经排班、被指定引用或非草稿状态的场次。
- 普通输入错误得到 422，业务冲突得到 409，前端能够展示失败信息。
- 所有新增行为均有自动化测试，且全量检查通过。
